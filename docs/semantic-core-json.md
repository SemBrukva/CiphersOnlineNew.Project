# Формат JSON для семантического ядра

Файлы semantic-core описывают осмысленное SEO-ядро конкретной страницы инструмента. Они лежат в:

`private/storage/semantic-core/<locale>/<category_alias>/<tool_alias>.json`

Пример:

`private/storage/semantic-core/ru/codes-and-alphabets/morse-code.json`

Этот JSON — curated source of truth для агентов, админки и будущей синхронизации в БД. Сырые выгрузки из внешних источников остаются в `private/storage/semantic-raw/`.

## Как создать файл

Сначала рядом должны лежать raw CSV и meta-файл:

```text
private/storage/semantic-raw/ru/codes-and-alphabets/morse-code.csv
private/storage/semantic-raw/ru/codes-and-alphabets/morse-code.meta.json
```

Импорт:

```bash
php bin/console semantic:raw:import private/storage/semantic-raw/ru/codes-and-alphabets/morse-code.csv --force
```

Проверка:

```bash
php bin/console semantic:validate
```

Синхронизация в БД:

```bash
php bin/console migrate
php bin/console semantic:sync
```

Для смысловой доработки используйте skill:

`docs/skills/semantic-core-builder/SKILL.md`

## Верхний уровень

```json
{
  "schema": "semantic-core.v1",
  "locale": "ru",
  "cluster": "азбука морзе переводчик",
  "intent": "tool",
  "status": "reviewed",
  "source": {},
  "tool": {},
  "analysis": {},
  "queries": [],
  "notes": "...",
  "curation": {}
}
```

### `schema`

Версия формата. Сейчас используется только:

```json
"semantic-core.v1"
```

### `locale`

Язык семантического ядра: `ru`, `en`, `de`, `es` и т. д. Должен совпадать с директорией файла.

### `cluster`

Название семантического кластера человеческим языком. Обычно это главный запрос или наиболее точная формулировка интента.

Пример:

```json
"азбука морзе переводчик"
```

### `intent`

Общий тип страницы. Для страниц инструментов обычно:

```json
"tool"
```

В будущем возможны другие значения: `category`, `article`, `landing`.

### `status`

Состояние curated JSON:

- `draft` — файл создан автоматическим импортом и ещё не проверен агентом/человеком;
- `reviewed` — файл смыслово проверен, приоритеты и рекомендации можно использовать;
- `mapped` — кластер полностью связан со страницей и готов к синхронизации/аналитике;
- `archived` — кластер больше не используется.

## `source`

Описывает исходный raw-сигнал.

```json
{
  "provider": "webmaster_yandex",
  "score_metric": "yandex_demand",
  "raw_file": "private/storage/semantic-raw/ru/codes-and-alphabets/morse-code.csv",
  "imported_at": "2026-06-25T21:56:00+02:00"
}
```

- `provider` — источник данных: `webmaster_yandex`, `google_ads`, `ahrefs`, `serpstat`, `manual` и т. п.
- `score_metric` — что означает `score` в этом источнике. Например, `yandex_demand`, `avg_monthly_searches_midpoint`, `manual_priority`.
- `raw_file` — путь к CSV, из которого создан JSON.
- `imported_at` — дата импорта или дата исходной выгрузки.

Важно: `score` не считается универсальной частотностью. Это источникозависимый вес запроса.

## `tool`

Связь семантики с инструментом и контентным JSON.

```json
{
  "slug": "codes-and-alphabets/morse-code",
  "url": "/ru/codes-and-alphabets/morse-code",
  "content_file": "private/storage/content/codes-and-alphabets/morse-code/ru.json"
}
```

- `slug` — внутренний путь инструмента без локали.
- `url` — публичный URL страницы.
- `content_file` — JSON контента, который агент должен учитывать при написании/редактировании страницы.

## `analysis`

Главный аналитический блок. Он нужен не для поисковиков напрямую, а для агента, редактора и админки: какие смыслы важны, какие интенты есть в кластере, что нужно покрыть на странице.

```json
{
  "score_metric": "yandex_demand",
  "total_score": 48862,
  "primary_terms": [],
  "modifiers": [],
  "intents": {},
  "content_recommendations": [],
  "query_groups": []
}
```

### `analysis.score_metric`

Дублирует или уточняет `source.score_metric`. Показывает, как читать числовой вес запросов.

Пример:

```json
"yandex_demand"
```

### `analysis.total_score`

Сумма `queries[].score`. Это не абсолютный спрос сайта, а общий вес кластера внутри текущего источника.

Используется для сравнения кластеров из одного и того же источника/метрики. Не стоит напрямую сравнивать `yandex_demand` с Google Ads range midpoint или ручной оценкой.

### `analysis.primary_terms`

Список значимых слов, которые чаще всего формируют кластер с учётом `score`.

Пример:

```json
[
  "морзе",
  "переводчик",
  "азбука",
  "перевод",
  "русский"
]
```

Назначение:

- быстро понять словарь кластера;
- подсказать агенту естественные слова для `meta_title`, intro, FAQ;
- увидеть странные или лишние темы, если в кластер попал мусор.

Это не список слов, которые надо механически вставить на страницу.

### `analysis.modifiers`

Устойчивые уточнения запросов.

Пример:

```json
[
  "на русский",
  "с русского",
  "онлайн",
  "по звуку",
  "со звуком"
]
```

Назначение:

- показать пользовательские сценарии;
- помочь выделить FAQ и дополнительные блоки;
- понять, нужны ли отдельные сервисы или текущая страница покрывает всё.

Для Morse-кластера модификаторы `на русский` и `с русского` показывают два направления одного инструмента, а `со звуком` — отдельный продуктовый сценарий внутри той же страницы.

### `analysis.intents`

Счётчики интентов по всем запросам.

Пример:

```json
{
  "cipher": 9,
  "online": 8,
  "encode": 3,
  "decode": 2
}
```

Текущие интенты:

- `tool` — общий инструментальный запрос без более точного интента;
- `cipher` — запрос явно связан с шифром, кодом или конкретной шифровальной системой;
- `translate` — запрос на переводчик/конвертер;
- `decode` — перевод из кода/шифра в читаемый текст;
- `encode` — перевод исходного текста в код/шифр;
- `audio` — звук, прослушивание, перевод по звуку;
- `online` — явное ожидание онлайн-инструмента;
- `english` — запросы с английским языком или латиницей.

Интенты помогают понять, что должно быть на странице. Например, если есть и `decode`, и `encode`, нельзя писать страницу только как “расшифровщик”. Если есть только `cipher` и `online`, это скорее запрос на онлайн-инструмент или симулятор, а не на текстовый переводчик.

### `analysis.content_recommendations`

Рекомендации для агента или редактора при создании/обновлении `tool.content_file`.

Пример:

```json
[
  "В meta_title сохранить точную связку «азбука морзе переводчик» и не размывать её справочной формулировкой.",
  "В первом экране явно назвать оба направления: перевод текста в азбуку Морзе и расшифровка Морзе обратно в русский текст."
]
```

Это не текст для прямой вставки на страницу. Это рабочие инструкции:

- какие формулировки сохранить;
- какие сценарии обязательно покрыть;
- какие темы держать в FAQ;
- чего избегать в title/H1.

### `analysis.query_groups`

Смысловые группы внутри кластера. Это самый важный блок для понимания структуры семантики.

```json
{
  "name": "Декодирование на русский",
  "intent": ["translate", "decode"],
  "priority": "secondary",
  "target": "intro",
  "representative_queries": [
    "переводчик с азбуки морзе на русский",
    "перевод с азбуки морзе на русский"
  ]
}
```

Поля группы:

- `name` — человекочитаемое название группы;
- `intent` — какие интенты покрывает группа;
- `priority` — важность группы для страницы;
- `target` — где лучше покрыть группу;
- `representative_queries` — несколько показательных запросов, не обязательно полный список.

Группы нужны, чтобы агент не пытался вставить 72 запроса подряд. Он должен писать страницу по группам смыслов: главный переводчик, декодирование, кодирование, звук, английский и т. д.

## `queries`

Полный список запросов после сортировки и смысловой разметки.

```json
{
  "query": "переводчик с азбуки морзе на русский",
  "score": 1511,
  "competitiveness": "AVERAGE",
  "priority": "secondary",
  "intent": ["translate", "decode"],
  "target": "intro"
}
```

### `queries[].query`

Исходная поисковая фраза.

### `queries[].score`

Вес запроса из raw-источника. Смысл зависит от `source.score_metric`.

### `queries[].competitiveness`

Оценка конкуренции из источника. Может быть пустой, если источник её не даёт.

Примеры:

- `LOW`
- `AVERAGE`
- `HIGH`

### `queries[].priority`

Приоритет запроса:

- `primary` — главный запрос страницы. Обычно один.
- `secondary` — важная формулировка или важный сценарий.
- `long_tail` — низкочастотный/уточняющий запрос для FAQ, примеров или второстепенного текста.
- `excluded` — запрос не должен покрываться этой страницей, но оставлен для истории решения.

### `queries[].intent`

Список интентов конкретного запроса. Один запрос может иметь несколько интентов:

```json
["translate", "decode", "online"]
```

### `queries[].target`

Рекомендуемая зона покрытия:

- `meta_title` — SEO title;
- `h1` — главный заголовок, если нужно отдельно от title;
- `intro` — первый экран/описание инструмента;
- `tool_ui` — подписи интерфейса;
- `content_block` — основной информационный блок;
- `faq` — FAQ;
- `examples` — примеры использования;
- `ignore` — не покрывать в контенте.

## `curation`

След ручной или агентной проверки.

```json
{
  "reviewed_at": "2026-06-26",
  "reviewed_by": "semantic-core-builder",
  "decision": "single_tool_cluster",
  "notes": []
}
```

- `reviewed_at` — дата смысловой проверки;
- `reviewed_by` — кто или какой skill проверял файл;
- `decision` — решение по кластеру;
- `notes` — почему принято такое решение.

Примеры `decision`:

- `single_tool_cluster` — всё покрывает одна страница инструмента;
- `split_recommended` — часть запросов лучше вынести в отдельный инструмент/страницу;
- `needs_review` — нужна ручная проверка;
- `low_priority` — спрос есть, но развитие не приоритетно.

## `notes`

Свободное поле для краткой общей заметки. Его можно использовать, чтобы объяснить агенту контекст, который не помещается в строгие поля.

## Правила редактирования

1. Не менять `schema`, если формат файла реально не менялся.
2. Не превращать `score` в `volume`: это разные понятия.
3. После автоматического импорта оставлять `status: draft`, после смысловой проверки менять на `reviewed`.
4. `primary` обычно должен быть один. Если кажется, что их несколько, возможно, кластер надо разделить.
5. Не вставлять все запросы в контент дословно. Контент пишется по `query_groups`, `intent` и `content_recommendations`.
6. Запросы с низким `score`, но важным продуктовым сценарием, можно оставлять в `long_tail` и покрывать через FAQ.
7. Запросы, которые указывают на отдельный сервис, лучше помечать как `excluded` или описывать в `curation.notes`.
8. Перед использованием файла в админке или синхронизацией с БД запускать `php bin/console semantic:validate`.

## Связь с БД

Semantic-core JSON остаётся источником истины для curated-семантики. БД должна использоваться для:

- фильтров и поиска в админке;
- связи с позициями;
- Search Console / Webmaster метрик;
- CTR, показов, кликов;
- истории изменений и снапшотов.

Динамические метрики не пишутся обратно в semantic-core JSON.

Текущие таблицы:

- `semantic_clusters` — один ряд на semantic-core кластер;
- `semantic_queries` — запросы кластера с `score`, `priority`, `intent`, `target`;
- `semantic_rank_snapshots` — будущие снимки позиций и поисковых метрик по запросам.

Команда `semantic:sync` читает JSON из `private/storage/semantic-core/`, обновляет кластер и пересоздаёт его запросы в БД. Снимки позиций не затрагиваются.
