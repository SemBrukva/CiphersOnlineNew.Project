# P1 — Масштабирование архитектуры инструментов шифрования/декодирования

Дата создания: 2026-05-27.
Статус: завершён (Этапы 1-5 выполнены).

## Почему это приоритет P1

Сейчас проект уже поддерживает:
- 4 API-шифра (Цезарь, Плейфер, Бофор, Гронсфельд),
- 7 browser-декодеров (Base64, Hex, URL, Binary, ASCII, JWT, Unicode).

Текущая реализация рабочая, но в коде уже есть ручные маппинги и дублирование, которые кратно увеличат стоимость добавления новых инструментов при росте до 30-50 сервисов.

---

## Проблемы, зафиксированные аудитом

### 1) Дублирование API-пайплайна в `GuestController`

Симптом:
- Методы `caesar()`, `playfair()`, `beaufort()`, `gronsfeld()` повторяют один и тот же каркас: JSON payload → валидация direction/text/settings → auto alphabet → process → JSON response.

Риск:
- Каждый новый API-инструмент будет добавлять копипасту и расхождение по поведению/ошибкам.

### 2) Ручные slug-маппинги в `CipherController`

Симптом:
- `examplesByTool`, `apiActionByTool`, `buildToolSettings()` завязаны на большие массивы и `match`.

Риск:
- Каждое добавление/переименование alias требует правок в нескольких местах.
- Выше шанс пропустить один из маппингов и получить «немой» баг в UI.

### 3) Дублирование алфавитов и общих операций в сервисах шифров

Симптом:
- Во всех 4 сервисах дублируются `ALPHABETS`, `detectAlphabet()`, `hasAlphabetCharacters()`.

Риск:
- Несогласованность при правках.
- Рост времени на поддержку и покрытие тестами.

### 4) Монолитный `cipher-tool.js` с ветвлением по slug

Симптом:
- `transform()` и `looksLikeEncoded()` завязаны на цепочки `if`/`flags`.

Риск:
- Файл быстро превратится в «точку конфликтов» при параллельной разработке.
- Сложнее писать изолированные тесты на отдельные декодеры.

### 5) Нестабильный контракт ошибок API ↔ UI

Симптом:
- API возвращает структуру `error: { code, message, ... }`, но UI местами берёт `error` как строку.

Риск:
- Пользователь видит некорректные сообщения.
- Сложно стандартизировать обработку ошибок для всех инструментов.

---

## Целевая архитектура

1. Единый `ToolRegistry` (slug/alias/tool id → метаданные, режим, settings provider, examples, api action).
2. Единый контракт API-инструмента (`validate + process + response meta`) и dispatcher в контроллере.
3. Общий компонент алфавитов (`AlphabetCatalog`/`AlphabetHelper`) для API-шифров.
4. Декомпозиция фронтенд-трансформеров по модулям + реестр трансформеров.
5. Унифицированный формат ошибок для UI (`error.message` как основная строка для показа).

---

## План реализации

### Этап 1. Реестр инструментов и исключение ручных маппингов (без изменения поведения)

Задачи:
1. Ввести `ToolRegistry` в `private/app/Cipher/` с источником данных для:
   - примеров (`example chips`),
   - `apiAction`,
   - `calculationMode`,
   - provider настроек UI.
2. Перевести `CipherController::buildToolUi()` и `buildToolSettings()` на реестр.
3. Сохранить обратную совместимость slug-алиасов (включая русские alias).

Готово, когда:
- В `CipherController` нет больших хардкод-словарей `examplesByTool` и `apiActionByTool`.
- Все текущие инструменты рендерятся как раньше.

### Этап 2. API dispatcher + общий контракт инструмента

Задачи:
1. Ввести интерфейс `ApiCipherToolInterface`.
2. Сделать адаптеры для 4 текущих шифров.
3. Добавить dispatcher в `GuestController` и заменить отдельные методы шифров единым обработчиком (или тонкими прокси).

Готово, когда:
- Добавление нового API-шифра требует:
  - зарегистрировать tool-класс,
  - добавить маршрут (или запись в конфиг),
  - без копипасты валидации в контроллере.

### Этап 3. Обобщение алфавитов

Задачи:
1. Вынести общий каталог алфавитов.
2. Вынести общие операции `supportedAlphabetCodes()`, `detectAlphabet()`, `hasAlphabetCharacters()`.
3. Упростить текущие сервисы шифров до алгоритм-специфичной логики.

Готово, когда:
- Данные алфавитов определены в одном месте.
- Все 4 сервиса используют общие операции через композицию/trait.

### Этап 4. Рефактор фронтенд-декодеров

Задачи:
1. Вынести преобразователи в отдельные модули (`decoders/base64.js`, `decoders/hex.js`, ...).
2. Ввести registry `toolSlug -> { transform, looksLikeEncoded }`.
3. Оставить `cipher-tool.js` как orchestration слой (UI + API вызовы).

Готово, когда:
- Добавление нового browser-декодера не требует менять большую цепочку `if`.
- Поведение текущих 7 декодеров покрыто тестами.

### Этап 5. Стандартизация ошибок API/UI

Задачи:
1. Зафиксировать контракт: пользовательское сообщение всегда в `error.message`.
2. Обновить фронтенд-обработку ошибок на чтение `error.message`.
3. Добавить тесты на вывод ошибок в UI.

Готово, когда:
- Ошибки всех API-инструментов показываются единообразно.

---

## Риски и ограничения

1. Нельзя ломать текущие публичные URL и alias.
2. Нельзя менять формат успешного ответа API без миграции фронтенда.
3. Любой рефактор должен быть покрыт unit/integration-тестами до удаления старых веток.
